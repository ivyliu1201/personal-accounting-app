# Step Confirmation Skill

## 1. 目的

本文件定義 AI agent 的小步驟開發節奏，確保每一步都有明確目標、事前確認、完成回報與使用者確認。

本文件只規範「開發節奏與交付流程」；測試選擇、測試執行與測試報告格式，依專案已啟用的測試回報 skill 或測試規則文件定義。

## 2. 核心原則

- 採 MVP 優先策略。
- 先做最小可行版本，再逐步擴充。
- 不要求一次做到位。
- 每一步只做一個小功能或一個明確修改範圍。
- 不做與本步目標無關的重構、擴充或格式調整。
- 不得自行補需求。
- 若發現需求、技術限制或文件規則衝突，必須回報，不得自行決定。

## 3. 實作前確認規則

每一步開始實作前，AI agent 必須先回報本步計畫，並取得使用者確認。

未取得使用者確認前，不得開始修改程式碼、設定檔、資料庫 schema、文件或依賴。

事前回報至少包含：

- 本步目標
- 預計修改範圍
- 預計修改檔案
- 可能影響範圍
- 是否涉及資料庫
- 是否涉及 migration
- 是否涉及環境變數
- 是否涉及文件更新
- 是否新增或調整第三方套件
- 本步不處理的範圍

## 4. 實作中規則

- 僅能修改事前回報中列出的範圍。
- 若實作過程中發現需要擴大範圍，必須停止並回報。
- 若遇到未定義行為，必須：
  - 在程式中標記 `TODO`
  - 在完成後回報中列出
  - 以最小可行且不違反既有文件的方式暫時處理
- 若發現文件衝突，必須回報待確認事項，不得自行選邊。
- 若發現目前技術棧無法支援需求，必須回報阻塞點與建議方案，不得自行修改需求。

## 5. 完成後回報規則

每一步完成後，AI agent 必須回報結果，並等待使用者確認後才能進入下一步。

完成後回報至少包含：

- 變更摘要
- 受影響檔案
- 已完成項目
- 部分完成項目
- 未完成項目
- 資料庫變更
- migration 變更
- 環境變數變更
- 第三方套件變更
- 文件變更
- TODO 清單與優先級
- 風險 / 待確認事項與優先級
- 測試報告或測試摘要

若專案已啟用測試回報 skill，測試內容必須依該 skill 的格式提供。  
若專案未啟用測試回報 skill，至少需列出：

- 測試項目
- 測試指令
- 測試結果
- 未測項目與原因

## 6. Git 操作規則

- 每一步開始前，AI agent 應先確認目前工作區狀態。
- 若工作區已有使用者未提交變更，必須停止並回報，不得覆蓋或混入該變更。
- 每一步實作完成，且完成後回報與測試 / 檢查結果已提供後，需詢問使用者是否建立 commit。
- 未取得使用者確認前，不得自行建立 commit。
- commit 必須只包含本步驟相關變更，不得混入下一步或無關修改。
- commit message 需簡短描述本步完成內容。
- 若本步測試失敗、存在未完成項目或仍有高優先級阻塞事項，不得建立 commit，需先回報原因與建議下一步。

## 7. 等待確認規則

完成後必須等待使用者確認，才能進入下一步。

使用者若提出修改意見，AI agent 必須將該意見視為下一步目標，不得直接延伸到其他未要求範圍。

若使用者明確要求一次處理多個項目，AI agent 仍需將範圍拆成清楚的子項目並回報影響範圍。

## 8. 錯誤與阻塞回報

若遇到錯誤或卡住，AI agent 必須停止擴大修改，並回報目前狀態。

錯誤與阻塞回報至少包含：

- 錯誤現象
- 可能原因
- 已嘗試處理方式
- 目前阻塞點
- 影響範圍
- 建議下一步

## 9. TODO 與風險分級

TODO 清單需標示優先級：

- 高
- 中
- 低

風險 / 待確認事項也需標示優先級：

- 高
- 中
- 低

分級原則：

- 高：會阻塞功能完成、造成資料錯誤、影響安全性，或導致主要流程無法使用。
- 中：不阻塞主要流程，但會影響使用體驗、維護性或後續擴充。
- 低：不影響主要流程，可延後處理的改善、整理或補強事項。
