# Frontend UI Rules Skill

## 1. 目的

本文件定義前端 UI、響應式版面、互動狀態與基本可用性的共用規則。

除非使用者在專案文件中另有指定，AI agent 在進行前端頁面、元件、版面調整、互動狀態或響應式設計時，應以本文件作為預設規則。

本文件的重點是讓同一個 Web App 在桌機與手機瀏覽時，都能保持清楚、可閱讀、可操作；但本文件不取代產品需求、技術棧、測試報告格式或開發節奏。

## 2. 適用範圍

本文件適用於以下工作：

- 前端頁面切版與版面調整
- 元件排列、間距、對齊與視覺一致性
- 桌機版與手機版的響應式處理
- loading、空狀態、成功提示、失敗提示等 UI 狀態呈現
- 表單、列表、表格、圖表、彈窗與導覽列的前端呈現
- 不改變產品功能前提下的前端可用性改善

本文件不適用於以下工作：

- 新增 `SPEC.md` 未定義的產品功能
- 修改資料模型、商業邏輯、API 契約或資料計算規則
- 新增外部服務或重大前端依賴
- 推翻 `SPEC.md`、`FLOW.md` 或 `WIREFRAME.md` 中已明確定義的需求

## 3. 與其他文件的關係

- 產品功能、頁面結構與互動需求，依 `SPEC.md` 為準。
- 使用者流程與狀態轉換，依 `FLOW.md` 輔助理解。
- 低保真版面、區塊位置與畫面資訊，依 `WIREFRAME.md` 輔助理解。
- 前端技術棧、專案初始化、依賴管理與建置方式，依 `skills/full-stack-java-spring.md`。
- 開發節奏與交付回報，依 `skills/step-confirmation.md`。
- 測試選擇、測試執行與測試報告格式，依 `skills/test-reporting.md`。

若本文件與 `SPEC.md` 衝突，以 `SPEC.md` 為準。

若本文件與 `WIREFRAME.md` 在版面細節上不一致，且不影響產品功能，AI agent 不得直接視為產品需求衝突；應先判斷是否屬於響應式或可用性調整，並在交付回報中說明調整原因與影響範圍。

若本文件與其他 skill 發生未定義的衝突，必須回報待確認事項，不得自行選邊。

## 4. 前端實作判斷順序

AI agent 處理前端 UI 任務時，應依下列順序判斷：

1. 先確認 `SPEC.md` 是否定義功能、欄位、操作與限制。
2. 再確認 `FLOW.md` 是否補充使用流程或狀態轉換。
3. 再確認 `WIREFRAME.md` 是否提供頁面區塊、排列或視覺參考。
4. 若文件未定義細節，才套用本文件的前端 UI 預設規則。
5. 若仍無法判斷，依 `skills/step-confirmation.md` 回報待確認事項。

不得以「前端最佳化」為理由，刪除、合併或改變 `SPEC.md` 定義的必要功能、欄位與主要操作。

## 5. 裝置與 viewport 判斷原則

本文件中的「手機版」與「桌機版」以瀏覽器 viewport 寬度作為主要判斷依據，不以裝置品牌、作業系統或 user-agent 作為主要判斷依據。

除非專案文件另有指定，預設採用以下檢查基準：

- 手機版：viewport 寬度小於 `768px`。
- 平板或中間寬度：viewport 寬度介於 `768px` 至 `1023px`。
- 桌機版：viewport 寬度大於或等於 `1024px`。

上述數值是本專案的預設檢查基準，不代表所有版面都必須只在這些寬度才改變。若內容在其他寬度已經出現擁擠、溢出或不可操作，AI agent 應依實際內容調整 breakpoint，並在交付回報中說明原因。

除非需求明確要求偵測實體裝置，否則不得僅依 user-agent 判斷使用者是手機或電腦；應優先使用響應式 CSS、容器寬度、media query 或等效方式處理不同 viewport。

## 6. 響應式設計原則

- 前端頁面預設需支援響應式設計。
- 主要頁面至少需在桌機版與手機版 viewport 下保持可閱讀、可操作。
- 響應式調整不得改變 `SPEC.md` 定義的核心功能、資料欄位與主要操作流程。
- 若 `WIREFRAME.md` 只提供桌機版線框，AI agent 可在不改變功能的前提下調整手機版排列，但不得自行刪除功能或改變主要互動。
- 若某個複雜版面在 MVP 階段無法完整支援手機版，必須回報為待確認事項或未驗證項目，不得描述為已完整完成。
- 響應式調整應優先處理可讀性、操作可點擊性、資訊完整性與避免橫向溢出。

## 7. 手機版版面規則

- 手機版下，主要區塊可由左右排列改為上下排列。
- 手機版下，內容應避免超出 viewport 造成非預期橫向捲動；若表格或圖表必須橫向捲動，需保留清楚的操作方式。
- 表單、列表、圖表與按鈕在手機版下不得造成主要操作無法使用。
- 互動按鈕在手機版下需保持可點擊，不得過度擁擠或被其他區塊遮擋。
- 導覽列在手機版下可改為折疊、換行或簡化排列，但不得移除主要頁面入口。
- 若手機版版面需要與 `WIREFRAME.md` 的桌機排列不同，需在交付回報中說明調整原因。

## 8. 桌機版版面規則

- 桌機版下，應優先維持 `WIREFRAME.md` 所定義的主要區塊位置與頁面結構。
- 桌機版可使用多欄、左右排列或較完整的資訊密度，但不得造成主要操作難以辨識。
- 若手機版與桌機版使用不同排列，兩者應保留相同核心資訊與主要操作。
- 不得只完成手機版而讓桌機版出現明顯破版、對齊錯誤或操作不可用。

## 9. 表格、列表與圖表規則

### 9.1 表格與列表

- 列表在手機版下可改為卡片式呈現，但需保留 `SPEC.md` 定義的主要欄位與操作。
- 若表格欄位過多，可採用卡片式排列、欄位換行、橫向捲動或精簡顯示。
- 不得隱藏 `SPEC.md` 規定的必要資訊。
- 分頁、篩選、類型切換、排序等控制元件需在手機版與桌機版下保持可操作。
- 若採用精簡顯示，必須在交付回報中列出被精簡的欄位、原因與使用者是否仍能查看完整資訊。

### 9.2 圖表

- 圖表需在容器寬度變化時保持可讀。
- 圖表在手機版下可縮小、上下排列或調整 legend / tooltip 呈現方式。
- 圖表響應式調整不得改變資料來源、資料計算邏輯或指標定義。
- 若圖表於手機版下需要簡化，必須在交付回報中說明簡化內容與影響範圍。

## 10. 表單與彈窗規則

- 表單欄位在手機版下可改為單欄排列。
- 表單送出、取消、刪除等主要操作按鈕需清楚可見。
- 表單錯誤訊息需出現在相關欄位附近，或使用者能明確知道是哪個欄位需要修正。
- 彈窗在手機版下不得超出可視範圍。
- 彈窗內容過長時需可捲動，且主要操作按鈕仍需可操作。
- 刪除確認、錯誤提示與成功提示在手機版下需保持可讀。

## 11. 狀態呈現規則

- loading、空狀態、成功提示與失敗提示需在相關區塊附近呈現。
- 狀態呈現不得遮擋主要操作，除非該操作在 loading 期間不可使用。
- 若資料更新只影響單一區塊，優先使用區塊級 loading，不要讓整頁無法操作。
- 錯誤訊息需清楚指出使用者可以如何修正。
- 若錯誤無法由使用者自行修正，需顯示可理解的失敗原因，避免只顯示技術性錯誤內容。
- 空狀態需說明目前沒有資料，並在適合時提供下一步操作入口。

## 12. 可用性與一致性原則

- UI 應以清楚、可操作、可維護為優先。
- 不得為了視覺效果犧牲可讀性或主要操作。
- 同一類型操作在不同頁面應盡量保持一致，例如新增、編輯、刪除、確認、取消。
- 顏色、間距、字級與按鈕樣式應保持一致，避免同一功能在不同頁面呈現差異過大。
- 主要操作與危險操作需在視覺上可區分。
- 若需要新增 UI 元件庫、圖表套件或重大前端依賴，必須依專案技術規則先回報原因與影響範圍，並取得使用者確認。

## 13. 不得自行決定的事項

AI agent 不得僅依本文件自行決定以下事項：

- 新增 `SPEC.md` 未定義的頁面、功能或主要操作流程
- 移除 `SPEC.md` 已定義的欄位、按鈕、頁面入口或資料呈現
- 改變資料計算邏輯、資料欄位意義或 API 契約
- 將手機版簡化內容描述為完整功能
- 為了 UI 美觀而修改產品規則
- 在未確認前新增重大前端依賴或外部服務

## 14. 驗證與回報要求

若本步涉及前端 UI、元件或響應式調整，完成後回報需包含：

- 調整的頁面或元件
- 主要版面變更
- 桌機版影響
- 手機版影響
- 是否影響 `SPEC.md` 定義的功能
- 已驗證的 viewport 或畫面寬度
- 已驗證的畫面或互動
- 未驗證項目與原因
- 若有簡化顯示，需說明簡化內容與使用者是否仍能取得完整資訊

若沒有實際執行畫面驗證，必須明確標示為未驗證，不得描述為已驗證。

除非專案文件另有指定，若要宣稱已完成響應式驗證，至少需檢查：

- 一個手機版 viewport，例如 `375px` 或 `390px` 寬度。
- 一個桌機版 viewport，例如 `1024px` 或更寬。

若只能檢查程式碼而無法開啟畫面，必須回報「僅完成程式碼層級檢查，尚未完成畫面驗證」。
