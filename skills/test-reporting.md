# Test Reporting Skill

## 1. 目的

本文件定義測試選擇、測試執行與測試報告格式，確保每次交付都有可追蹤、可驗證的品質回報。

本文件只規範測試相關內容；開發節奏、事前確認、完成後是否等待使用者確認，依 step-confirmation 或專案入口文件定義。

## 2. 測試原則

- 核心業務邏輯與資料處理採 TDD。
- UI 不強制全面 TDD，但重要互動流程需透過可執行測試或人工驗證回報覆蓋。
- build 必須通過。
- 測試必須通過。
- 若專案有 lint，lint 必須通過。
- 若有資料庫 migration，需確認 migration 可正常執行。
- 不得因測試失敗而直接移除測試，除非先回報原因並取得使用者確認。
- 不得將未執行的測試描述為已通過。
- 不得將失敗測試描述為通過。

## 3. 測試範圍判斷

每次交付前，AI agent 需依本次變更範圍選擇測試項目。

常見對應規則：

- 後端業務邏輯：單元測試優先。
- API 行為：Controller / API 測試或等價測試。
- 資料庫行為：Repository / migration / 整合測試。
- 前端資料處理：單元測試或元件測試。
- 前端 UI 行為：元件測試、端對端測試或人工驗證回報。
- 設定檔或啟動方式：驗證 build、啟動或設定載入行為。
- 文件或純文字調整：可不執行程式測試，但需說明未測原因。

若本次變更影響多個層級，測試範圍需涵蓋主要受影響層級。

## 4. 測試執行要求

交付前應執行與本次變更相關的測試。

若無法執行測試，必須在測試報告中說明：

- 未執行的測試項目
- 未執行原因
- 可能風險
- 建議補測方式

若測試失敗，必須依本文件第 6 節回報，不得略過或隱藏。

## 5. 測試報告格式

每次交付的測試報告至少包含：

- 測試項目
- 測試指令
- 通過數量
- 失敗數量
- 略過數量
- 覆蓋到的功能範圍
- 失敗原因
- 未測項目與原因
- 執行結果摘要

## 6. 測試報告範本

```md
## 測試報告

### 測試項目
- [ ] build
- [ ] unit tests
- [ ] integration tests
- [ ] lint
- [ ] manual verification

### 測試指令
```bash
<command>```

### 執行結果
通過：<number>
失敗：<number>
略過：<number>

### 覆蓋範圍
<本次測到的功能或模組>

### 失敗原因
<若無失敗，填寫「無」>

### 未測項目與原因
<若皆已測，填寫「無」>

###摘要
<簡短描述整體結果>
```

## 7. 測試失敗處理

若測試失敗，交付回報必須包含：

- 失敗測試名稱
- 錯誤訊息摘要
- 可能原因
- 已嘗試處理方式
- 是否阻塞交付
- 建議下一步

若測試失敗會影響主要功能、資料正確性、安全性或 build 結果，必須標示為阻塞交付。

## 8. 人工驗證回報規則

若使用人工驗證，需明確列出：

- 驗證項目
- 驗證步驟
- 預期結果
- 實際結果
- 是否通過

人工驗證不得取代應有的核心業務邏輯單元測試，但可用於補充 UI 流程、視覺狀態或第三方登入等不易單元測試的情境。
